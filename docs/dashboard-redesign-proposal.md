# OpenClaw Health Monitor - Dashboard 重构设计方案

## 1. 当前问题诊断

### 1.1 核心问题

经过深度分析，当前Dashboard存在以下结构性问题：

| 问题类别 | 具体表现 | 影响 |
|---------|---------|------|
| **信息过载** | 4个Tab，10+页面，50+卡片 | 用户找不到重点 |
| **层级混乱** | Tab、Page、Section、Card四级嵌套 | 认知负担重 |
| **重复内容** | "最近异常"出现3次，"环境状态"出现2次 | 维护困难 |
| **视觉平等** | 所有卡片视觉权重相同 | 无法区分优先级 |
| **交互分散** | 高危操作和普通查看混在一起 | 误操作风险 |
| **实时性差** | 所有数据5秒统一刷新 | 不必要的API压力 |
| **数据耦合** | `/api/status`返回所有数据 | 加载慢 |

### 1.2 与优秀产品的差距

| 维度 | DataDog/Grafana | 当前Dashboard |
|------|----------------|---------------|
| 信息架构 | 3层：Service → Overview → Detail | 4层：Tab → Page → Section → Card |
| 视觉层次 | Hero(健康评分) + KPI(3-5个) + Detail | 平铺50+个同等卡片 |
| 交互模式 | 查看/探索/管理分离 | 全部混在一起 |
| 数据加载 | 按需加载，分级刷新 | 全部5秒统一刷新 |
| 告警设计 | 红色=立即处理，黄色=关注，绿色=正常 | 颜色使用不一致 |
| 操作保护 | 高危操作二次确认 + 审计 | 直接执行 |

---

## 2. 重构目标

### 2.1 设计原则

基于`docs/dashboard-layout-redesign-brief.md`的要求，新设计遵循：

1. **单一主导叙事** - 第一屏必须回答：系统健康吗？有问题吗？该做什么？
2. **渐进式披露** - 次要信息默认折叠，需要时展开
3. **操作分层** - 查看、探索、管理、高危操作分离
4. **数据按需** - 不同数据不同刷新频率
5. **视觉权重** - 健康状态占50%视觉权重，其他50%

### 2.2 成功标准

- 新用户5秒内理解系统状态
- 健康/异常/晋升状态一目了然
- 操作员能清楚区分监控、动作、排障
- 页面在MacBook Air上流畅运行

│   ├── 运行验收          │   ├── 3个核心指标
│   ├── 任务链路          │   ├── 下一步行动
│   └── 代理活动          │   └── 最近事件时间线
├── 学习中心              │

### 4.2 Hero区域详细设计

#### 4.2.1 健康评分（Health Score）

**设计灵感**：DataDog的Service Health Score + Vercel的Status Indicator

```
健康评分算法（建议）：
- 基础分：100
- 环境健康：-20（如果gateway不健康）
- 任务阻塞：-10（每个阻塞任务）
- 内存压力：-10（如果>80%）
- CPU压力：-10（如果>90%）
- 错误日志：-5（每个错误分类）
- 学习滞后：-5（如果learning停滞）

分数区间：
- 90-100：优秀 🟢
- 70-89：良好 🟡
- 50-69：警告 🟠
- 0-49：严重 🔴
```

**UI组件**：
- 大号数字显示（48px）
- 环形进度条（颜色随分数变化）
- 点击查看评分详情（弹窗展开）

#### 4.2.2 下一步行动（Next Action）

**设计原则**：只显示最重要的1个行动建议

```
优先级排序：
1. 🔴 立即处理：环境不健康 + 处理按钮
2. 🟠 今天处理：有晋升准备 + 晋升按钮
3. 🟡 本周关注：学习滞后 + 查看按钮
4. 🟢 保持现状：系统健康 + 无需操作
```

## 7. 数据架构优化

### 7.1 API重构

从1个巨型API拆分为按需加载：

```python
# 当前：一个API返回所有数据
/api/status  →  返回 20+ 个数据对象

# 新架构：按需加载
/api/health-score       → 健康评分（5秒刷新）
/api/environment        → 环境状态（10秒刷新）
/api/tasks              → 任务列表（默认不刷新，手动刷新）
/api/agents             → 代理列表（30秒刷新）
/api/learnings          → 学习数据（60秒刷新）
/api/events             → 事件时间线（5秒刷新）
/api/metrics            → 系统指标（5秒刷新）
```

### 7.2 数据刷新策略

| 数据类型 | 刷新频率 | 触发条件 |
|---------|---------|---------|
| 健康评分 | 5秒 | 定时 + 用户手动 |
| 系统指标(CPU/内存) | 5秒 | 定时 |
| 事件时间线 | 5秒 | 定时 + 新事件 |
| 环境状态 | 10秒 | 定时 |
| 代理活动 | 30秒 | 定时 |
| 任务列表 | 手动 | 用户操作 |
| 学习数据 | 60秒 | 定时 |
| 配置数据 | 手动 | 用户操作 |

### 7.3 前端状态管理

```javascript
// 建议引入轻量级状态管理
const store = {
  // 实时数据（自动刷新）
  realtime: {
    healthScore: {},
    metrics: {},
    events: []
  },
  
  // 准实时数据（低频刷新）
  nearRealtime: {
    environment: {},
    agents: {}
  },
  
  // 静态数据（手动刷新）
  static: {
    tasks: {},
    learnings: {},
    config: {}
  },
  
  // 用户操作状态
  ui: {
    currentView: 'overview',
    selectedEnvironment: null,
    selectedTask: null,
    isLoading: false
  }
}
```
